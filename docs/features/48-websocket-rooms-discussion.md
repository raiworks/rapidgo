# 💬 Discussion: WebSocket Rooms / Channels

> **Feature**: `48` — WebSocket Rooms / Channels
> **Status**: 🟡 IN PROGRESS
> **Depends on**: Feature #31 (WebSocket Support) — ✅ SHIPPED
> **Date**: 2026-03-07

---

## What Problem Does This Solve?

Feature #31 provides the WebSocket upgrade primitive (`Upgrader`, `Handler`), but each connection is isolated — there is no built-in way to group connections or broadcast messages. Real-time applications (chat rooms, live dashboards, notifications, multiplayer games) require:

1. **Connection tracking** — know which connections are active
2. **Grouping** — organize connections into logical rooms/channels
3. **Broadcasting** — send a message to all connections in a room
4. **Targeted messaging** — send to a specific connection by ID

Without framework support, every application rebuilds the same hub/room plumbing from scratch. Feature #48 provides these primitives in `core/websocket/`.

---

## Scope

### In Scope

| Item | Description |
|------|-------------|
| Hub | Central registry of all active connections |
| Room | Named group of connections within the hub |
| Client | Wrapper around `*websocket.Conn` with a unique ID |
| Broadcast | Send a message to all clients in a room |
| Join/Leave | Add/remove a client from a room |
| Hub handler | Gin handler that upgrades and registers the client with the hub |

### Out of Scope

| Item | Rationale |
|------|-----------|
| Persistence | Room state is in-memory only — no DB/Redis backing |
| Authentication | Use existing auth middleware on the HTTP route before upgrade |
| Message serialization | App-level concern (JSON, protobuf, etc.) |
| Pub/Sub across processes | Single-process only — distributed pub/sub is a separate feature |
| GraphQL subscriptions | Separate concern — would layer on top of rooms |
| Message history/replay | App-level concern |

## Design Decisions

| # | Decision | Choice | Rationale |
|---|----------|--------|-----------|
| D1 | Package location | `core/websocket/` (same package) | Rooms extend the WebSocket primitive — no new package needed |
| D2 | Hub pattern | Central `Hub` struct with `sync.RWMutex` | Standard Go concurrency pattern; no channels needed for a map-based registry |
| D3 | Client identity | `Client` struct with string `ID` (UUID) | Unique per connection; generated by the framework |
| D4 | Room naming | Plain string keys | Simple, flexible — developer chooses naming convention |
| D5 | Broadcast message type | `[]byte` with `websocket.MessageType` | Matches the `coder/websocket` API — no opinion on payload format |
| D6 | Thread safety | `sync.RWMutex` on Hub | Read-heavy workload (broadcasts) benefits from RWMutex over plain Mutex |
| D7 | Client removal | Automatic on disconnect + explicit `Leave`/`Remove` | Hub detects closed connections during broadcast and cleans up |
| D8 | Existing code impact | Zero changes to `websocket.go` | New file alongside existing code — `Upgrader` and `Handler` remain unchanged |

## Open Questions

| # | Question | Resolution |
|---|----------|------------|
| Q1 | Should rooms be created explicitly or on-demand when a client joins? | ✅ On-demand — `Join` creates the room if it doesn't exist. `Leave` removes the room when empty. Keeps API minimal. |
| Q2 | Should the Hub expose a channel-based API or method-based API? | ✅ Method-based (`Join`, `Leave`, `Broadcast`). Channels add complexity without benefit for this use case — the Hub is a shared data structure, not a pipeline. |
| Q3 | Should `Broadcast` skip the sender? | ✅ No — `Broadcast` sends to ALL clients in a room. An `BroadcastOthers` variant sends to all except one client. Two methods are clearer than a flag. |
| Q4 | Should the Hub limit connections per room? | ✅ No — app-level concern. The framework provides the primitives. |
| Q5 | Should the Client buffer outgoing messages? | ✅ No — writes go directly to `conn.Write`. Buffering/queuing is app-level. Keeps the implementation simple. |

## Prior Art

| Framework | Pattern |
|-----------|---------|
| Gorilla WebSocket chat example | Hub struct with register/unregister channels, broadcast channel |
| Socket.IO (Node.js) | Rooms with join/leave, `io.to(room).emit()` |
| Phoenix (Elixir) | Channels with topics, join/leave callbacks |
| Laravel Broadcasting | Channels (public, private, presence) |

The RapidGo approach is closest to the Gorilla pattern but uses direct method calls instead of channels, and `sync.RWMutex` instead of a select loop. This is simpler and avoids goroutine-per-hub overhead.
