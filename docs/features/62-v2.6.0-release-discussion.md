# 💬 Discussion: v2.6.0 Release

> **Feature**: `62` — v2.6.0 Release
> **Date**: 2026-03-15
> **Status**: COMPLETE

---

## 1. What Are We Building?

A minor release (v2.6.0) delivering a `Logger` interface for pluggable logging backends, a lightweight typed config loader `LoadConfig[T]()`, and an additive `GenerateTokenFromString()` JWT helper. 3 core framework items. The 4th planned item (MOD-01 `rapidgo-modules/authstore`) is **deferred to v2.7.0** — it requires the `GenerateTokenFromString()` from this release and needs its own design cycle in a separate repo.

## 2. Why?

Three patterns emerged from production use and the SocioSky enhancement spec:

1. **Logger lock-in** — `core/logger.Setup()` returns `*slog.Logger` directly. There's no interface, so users can't swap in Zap, Zerolog, or a test mock without bypassing the framework entirely.
2. **Config boilerplate** — Every project manually calls `config.Env()`, `config.EnvInt()`, `config.EnvBool()` dozens of times to populate config structs. A generic `LoadConfig[T]()` eliminates this repetition using struct tags.
3. **UUID incompatibility** — `GenerateToken(userID uint)` hard-codes the user ID as `uint` in JWT claims. Projects using UUID primary keys have no framework-supported path to generate tokens.

## 3. Origin

Items cataloged in `docs/enhancements-improvements-bugs/index.md`. Cross-checked in `03-rapidgo-next-discussion.md`. All open questions resolved.

## 4. Scope

### Enhancements
- **ENH-06**: `Logger` interface in `core/logger/` — define `Logger` interface with `Debug/Info/Warn/Error` + `With(attrs)` methods. Default implementation wraps `*slog.Logger`. Framework internals use the interface. Users can plug in Zap/Zerolog via their own adapter and register it as a provider.
- **ENH-05**: Lightweight typed config `LoadConfig[T]()` in `core/config/` — generic function that reads env vars into a struct using `env:"KEY"` + `default:"value"` struct tags. Validates using `go-playground/validator`. ~150 LOC, no Viper.
- **ENH-02a**: Additive `GenerateTokenFromString(userID string)` in `core/auth/jwt.go` — same logic as `GenerateToken(uint)` but accepts string. Existing `uint` version marked deprecated but kept indefinitely.

### Out of Scope
- **MOD-01**: `rapidgo-modules/authstore` — deferred to v2.7.0. Needs its own repo setup, schema design, and the `GenerateTokenFromString()` from this release.
- Zap/Zerolog adapters in framework — users write their own adapter implementing `Logger` interface
- Viper config — decided against (too heavy). `LoadConfig[T]()` uses existing deps only.
- Breaking changes to `GenerateToken(uint)` — stays forever

## 6. Dependencies & Risk

| Risk | Mitigation |
|------|------------|
| Logger interface design could be too narrow or too wide | Model after `slog.Handler` — keep it minimal: `Debug/Info/Warn/Error` + `With`. Avoid `Printf`-style methods. |
| `LoadConfig[T]()` reflection could be slow | Only called once at startup. Performance is irrelevant. |
| `LoadConfig[T]()` requires `validator` as direct dep | Already indirect via gin. Make it direct in `go.mod`. |
| Existing code using `slog.Info()` globally | Framework internals must be updated to use the interface. Can be done incrementally — `slog.SetDefault()` still works as fallback. |

## 7. Discussion Resolution

All items verified against codebase. Architecture decisions finalized in `03-rapidgo-next-discussion.md`:
- ENH-05: No Viper — build `LoadConfig[T]()` with existing deps
- ENH-06: Interface in core, no Zap code in framework
- ENH-02a: Additive only, keep `uint` version deprecated indefinitely
- MOD-01: Deferred to v2.7.0
