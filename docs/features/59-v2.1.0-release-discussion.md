# 💬 Discussion: v2.1.0 Release

> **Feature**: `59` — v2.1.0 Release
> **Date**: 2026-03-11
> **Status**: COMPLETE

---

## 1. What Are We Building?

A minor release (v2.1.0) adding 2 new features, 4 improvements, and 4 documentation guides to the RapidGo framework. All items are backward-compatible — no breaking changes.

## 2. Why?

Building SpecialMedia on RapidGo surfaced gaps in the framework. These are features every app built on RapidGo would benefit from, not app-specific concerns. Shipping them as a framework release means all future projects get them too.

## 3. Origin

Items were cataloged in `docs/rapidgo-backlog.md` during SpecialMedia architecture analysis. Classified into framework vs project-level, with 7 items selected for framework.

## 4. Scope

### New Features
- **F-002**: Cursor-based pagination — `CursorPaginate()` helper alongside existing `Paginate()` (which is also being promoted into the framework)
- **F-004**: Notification system — `Notifiable` interface with channel dispatch (database, mail)

### Improvements
- **I-001**: WebSocket room cleanup — heartbeat/ping and disconnect cleanup
- **I-002**: Queue retry with configurable exponential backoff per job type
- **I-003**: Rate limiter per-user key strategy (not just per-IP)
- **I-005**: GORM query logging in development mode

### Documentation
- **D-001**: WebSocket rooms/channels usage guide
- **D-002**: Queue job creation and registration guide
- **D-003**: OAuth2 provider setup guide
- **D-004**: Multi-port serving (service mode) guide

### Out of Scope
- F-001 (image processing), F-003 (polymorphic helpers), F-005 (soft delete scopes), F-006 (admin improvements), I-004 (config hot-reload) — all project-level
- No breaking changes to existing APIs

## 5. Dependencies & Risk

| Risk | Mitigation |
|---|---|
| New `notification` package adds dependency coupling | Keep it optional — apps only import if they use it |
| Rate limiter changes could break existing usage | Additive only — existing `RateLimitMiddleware()` still works as-is |
| Queue backoff changes behavior | Only changes when `MaxAttempts > 1` and job actually fails |

## 6. Decision

Proceed. Build in dependency order: improvements first (small, isolated), then new features, then docs.
